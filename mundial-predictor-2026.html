Seis módulos funcionales independientes que se sirven desde una plataforma unificada. El núcleo ML opera sin dependencias externas; los módulos de enriquecimiento y experiencia de usuario se activan sobre él.

Resultados oficiales integrados · ELO recalculado partido a partido
Marcadores en tiempo real desde football-data.org (proxy cacheado en Next.js API Route). Cada resultado oficial actualiza automáticamente el veredicto del modelo: si acertó la predicción, qué probabilidad le asignó, y el delta de ELO post-resultado.

🎯
Predictor de partido
48 selecciones · Cualquier cruce · Narrator AI bogotano
Probabilidades victoria/empate/derrota del Ensemble para cualquier cruce entre las 48 selecciones del torneo. ELO actualizado diariamente con resultados reales. Probabilidades calibradas isotónicamente para que un 60% corresponda a 60% de frecuencia real.
Narrator AI: narración futbolera en español bogotano con contexto real del partido (tabla del grupo, localía, probabilidades, presión de clasificación). Pre-computada una vez al día — cero costo LLM por usuario para contenido cacheado desde narrations.json estático.
Fallback LLM automático: si una narración no existe en cache (partidos knockout no pre-computados), el sistema genera sobre la marcha sin degradar la experiencia del usuario. Hit rate objetivo: ≥95% desde JSON estático.

Chat IA con RAG
DeepSeek + contexto real del torneo · Caché SHA-256 · Rate limit 20 req/hr/IP
Preguntas sobre el torneo en lenguaje natural: partidos del día, tabla de grupos, predicciones específicas, historial de resultados. El sistema inyecta contexto real (standings actualizados, resultados, probabilidades del Ensemble) antes de cada llamada LLM.
RAG con embeddings DashScope (Qwen3 text-embedding-v3). Índice pre-construido en rag_index.json. Cache de respuestas con clave SHA-256 y TTL 2 horas, ~70-80% hit rate en instancias warm de Vercel.
Filtro de temas y rate limit de 20 req/hora/IP sin Redis (module-level Map). Temas fuera del scope del torneo son rechazados antes de consumir tokens. Fallback Anthropic Claude si DeepSeek no responde.
🤖
Agent Debate System
3 agentes expertos · Razonamiento lógico · 4 predicciones por partido
3 agentes debaten en 3 rondas usando DeepSeek Reasoner (extended thinking): Group Analyst (presión de clasificación + puntos + GD), Tactical Scout (estilos de juego + táctica), Sentiment Reader (momentum psicológico de MD1/MD2). No usan el modelo ML — razonan solo desde lógica de torneo.
4 predicciones por partido: 3 individuales (una por agente) + 1 consenso. Cada predicción incluye marcador esperado y probabilidad. Permite evaluar cuál tipo de razonamiento (clasificación, tácticas, sentimiento) es más predictivo históricamente.
Evaluación granular: fronted muestra precision individual por agente (1X2 y marcador exacto). Identifica qué razonador acierta más en contextos específicos. Aparece en el Predictor y en "En Vivo → Próximos".
Forward-only: debates se ejecutan manualmente para partidos puntuales antes de que se jueguen. No hay backfill retroactivo de partidos ya jugados. Acumulativo: nueva data se agrega sin pisar debates previos.

Núcleo ML — Ensemble
Determinístico · Zero dependencias externas · Siempre disponible · RPS 0.1958
Sistema ELO propio
K variable por torneo, margen de victoria, factor de localía. Calculado solo sobre resultados — sin sesgos de confederación. Peso: 22%.
Poisson Bivariado
Distribución de goles esperados por equipo. Señal de distribución de score que ni ELO ni XGB capturan individualmente. Peso: 58%.
XGBoost 2.0
Patrones no-lineales sobre feature matrix histórica. Complementa a Poisson en interacciones complejas. Peso: 20%.
Calibración Isotónica
Post-hoc sobre el Ensemble. Garantiza que las probabilidades sean confiables para simulación Monte Carlo: p=0.60 → 60% frecuencia real.
MLflow
Tracking de experimentos, métricas por run (RPS, log-loss, Brier, ECE), comparación de modelos y registro de artefactos serializados.
Monte Carlo (TypeScript)
5,000 simulaciones del torneo completo en el navegador. ~200ms en hardware moderno. Sin ninguna llamada al servidor. Seed determinístico (42).
scikit-learn
Pipeline de preprocessing, calibración isotónica, validación de modelos base, ablation testing de features candidatas.
scipy + statsmodels
Distribución de Poisson bivariada, cálculo de probabilidades de resultados exactos y distribuciones de goles.

Pipeline en vivo durante el torneo
MLOps & Ciclo Operativo
El sistema operó en producción durante 39 días (11 Jun – 19 Jul 2026). Ciclo diario automatizado con protocolo de doble corrida en MD2 y manejo especial de partidos simultáneos en MD3.
1
Fetch resultados del día anterior
live_update.py descarga marcadores oficiales de football-data.org e integra en wc2026_live_results.csv. Modo --dry-run disponible para preview sin escritura.
2
Recalcular ELO y re-entrenar Ensemble
ELO se recalcula cronológicamente sobre todos los resultados incluyendo los nuevos. El Ensemble se reentrena con features actualizadas. El modelo aprende de cada partido jugado. Tiempo: ~80s.
3
Predicciones live con anti-leakage por partido
predict_live.py --export genera probabilidades para cada partido pendiente con ELO cutoff individual (kickoff − 60s). Agentes opcionales si presupuesto disponible. Exporta a frontend/public/data/.
4
Pre-computar narraciones y previas de grupo
precompute_narrations.py genera narraciones para los partidos de hoy × dialectos activos (español bogotano en grupos). Fuerza-regenera narraciones del día actual y previas de grupo. Una llamada LLM por partido — cero por usuario.
5
Deploy a Vercel
npx vercel --prod. CDN global actualizado en ~30s. Dashboard muestra resultados reales, probabilidades actualizadas y narraciones del día.
🌅
Corrida matutina completa (igual que MD1)
Ciclo completo antes de los primeros partidos del día: fetch → ELO → retrain → predict → narrations → deploy. MD2 tiene 4 partidos/día split en ventanas tarde/noche.
⚽
Primeros 2 partidos se juegan (tarde)
Los resultados de los primeros 2 partidos del día cambian radicalmente la presión de clasificación para los equipos que juegan de noche. Sin actualización, las predicciones nocturnas ignoran esta presión real.
🔁
Segunda corrida (tarde): resultados → predicciones nocturnas
live_update.py → predict_live.py --export. Las predicciones de los partidos nocturnos incorporan presión real de puntos y diferencia de goles de los partidos de tarde. Tiempo: ~90s.
📖
Actualizar solo previas de grupo (segunda corrida)
precompute_narrations.py --groups-only --days 1. Las previas narrativas de los grupos con partidos nocturnos reflejan la presión real de clasificación de los equipos que ya jugaron. Costo: ~$0.005.
🚀
Deploy nocturno actualizado
Segundo deploy del día. El dashboard refleja resultados de tarde y predicciones con contexto real de clasificación para los partidos de noche. Usuarios que vean el predictor antes del partido nocturno ven probabilidades actualizadas.
⚡
Protocolo de partidos simultáneos
En MD3, todos los partidos de un grupo arrancan a la misma hora. Los standings se interpretan como distribuciones de probabilidad de escenarios, no como secuencia de resultados. El ELO cutoff es aún más crítico.
1
Corrida pre-MD3: ELO cutoff independiente por partido
predict_live.py calcula ELO cutoff para cada partido simultáneo de forma independiente. Partido A del Grupo X y Partido B del Grupo X tienen el mismo ELO base porque ninguno de los dos se ha jugado aún.
2
Análisis best-third y escenarios de clasificación
FIFA-Regs-Strategist proyecta cuáles equipos clasificarán como mejor tercero de cada grupo. Las previas narrativas enfatizan presión directa, diferencia de goles y escenarios de clasificación compartida entre grupos.
3
Post-MD3: deploy final de fase de grupos
Con los 72 partidos de grupos completados, se exportan los standings definitivos, los 32 clasificados por posición/grupo y la tabla del bracket de eliminatoria directa para la siguiente fase.

Principios de diseño FinOps
Core primero, agentes opcionales. El Ensemble predice con precisión completa (RPS 0.1958) sin ningún agente LLM. Los agentes son enriquecimiento experimental — su impacto no está medido históricamente.
Fallback graceful sin degradación. Si el presupuesto se agota durante el ciclo, el sistema cae automáticamente al Ensemble. El usuario no nota diferencia funcional — solo ausencia de delta de enriquecimiento.
Transparencia de deltas. El frontend siempre muestra el prior del Ensemble. Los deltas de los agentes están explícitamente labeled como "enriquecimiento opcional" — el usuario sabe qué es el modelo y qué es contexto adicional.
Narrator AI pre-computado = costo fijo. $0.015–$0.025/día independientemente del número de usuarios simultáneos. Sin sorpresas de costo a escala.
Orquestador como rate limiter natural. Máximo 2 agentes por consulta. Context stripping elimina fluff conversacional antes de cada llamada. Compresión de payload mediante tablas Markdown/JSON en lugar de prosa libre.

Reglas de negocio
Invariantes del sistema
Reglas que el sistema debe cumplir siempre, independientemente del contexto. Divididas por dominio: integridad del modelo, agentes, FinOps, frontend y operaciones. Violaciones de reglas críticas deben generar alerta inmediata o bloquear el pipeline.
Modelo & Predicción
RN-MOD-01 · CRÍTICA
Zero data leakage en entrenamiento
El conjunto de entrenamiento nunca puede contener datos posteriores al año de test. Train ≤ 2021. Test = Qatar 2022. Esta regla es verificada automáticamente por test_temporal_split_no_leakage — si falla, el pipeline se aborta.
RN-MOD-02 · CRÍTICA
Anti-leakage por partido en predicción live
Cada partido en vivo debe usar un ELO calculado con cutoff exacto de kickoff − 60 segundos. Nunca usar el ELO del fin del día. En partidos simultáneos (MD3), cada partido tiene su propio ELO independiente basado en el estado del mundo justo antes de su kickoff.
RN-MOD-03 · ALTA
Probabilidades siempre normalizadas a 1.0
El output final (p_home + p_draw + p_away) debe sumar exactamente 1.0 después de cada blend de Ensemble y después de cada ajuste de deltas de agentes. El Orchestrator es responsable de renormalizar antes de exportar.
RN-MOD-04 · ALTA
Ablation testing antes de incorporar nuevas features
Ninguna feature nueva puede agregarse a FEATURE_COLS sin demostrar mejora de RPS vs el base mediante ablation_features.py. Una feature que no mejora el RPS no se incorpora, aunque parezca lógica desde el dominio.
RN-MOD-05 · MEDIA
ELO propio, no de terceros
El sistema ELO se calcula internamente sobre los 49,765 partidos históricos con K variable por torneo y factor de margen de victoria. No se acepta importar ELO de fuentes externas (FIFA rankings, clubelo.com) — introduce sesgos no controlados.
Sistema Multi-Agente
RN-AGE-01 · CRÍTICA
Máximo 2 agentes por consulta de predicción
El Orchestrator nunca puede invocar más de 2 agentes especializados por partido. Esta regla controla el costo por llamada y mantiene el tiempo de respuesta bajo. Si el routing sugiere 3+ agentes, el Orchestrator prioriza los 2 de mayor impacto esperado.
RN-AGE-02 · CRÍTICA
Delta de cada agente clamped a ±12%
Ningún agente puede proponer un delta de probabilidad mayor a ±12% en ninguna dirección (home/draw/away). El Orchestrator clampea antes de aplicar. La suma de deltas también debe resultar en probabilidades no negativas para ninguno de los tres outcomes.
RN-AGE-03 · ALTA
FinOps-Market-Calibration NO es herramienta de apuestas
El agente de calibración de mercado detecta divergencias entre el modelo y las casas de apuestas — nunca recomienda apostar. El output es exclusivamente un signal de calibración (¿difiere el mercado del modelo?) para uso interno del pipeline, no para consumo directo del usuario.
RN-AGE-04 · ALTA
Context stripping obligatorio antes de llamadas LLM
El Orchestrator elimina fluff conversacional, turnos de contexto redundantes e información histórica no relevante para el partido específico antes de pasar el payload a cada agente. Solo se pasa el estado actual del partido y las variables necesarias para el cálculo.
RN-AGE-05 · MEDIA
Agentes determinísticos siempre corren, independiente del presupuesto
FinOps-Market-Calibration y FIFA-Regs-Strategist tienen costo $0 y no consumen tokens LLM. Deben ejecutarse en todas las predicciones, incluso cuando el budget guard ha bloqueado los agentes LLM. Su output enriquece el prior sin costo adicional.
Agent Debate System
RN-DEB-01 · CRÍTICA
4 predicciones por partido: 3 agentes individuales + 1 consenso
El JSON output debe siempre contener: group_analyst, tactical_scout, sentiment_reader, consensus. Cada predicción incluye home_goals, away_goals, probability. El consenso es el blend de los 3 individuales, no un cuarto agente independiente. Todas las 4 predicciones deben evaluarse contra el resultado real.
RN-DEB-02 · CRÍTICA
Forward-only: sin backfill retroactivo
Los debates se ejecutan manualmente para partidos puntuales ANTES de que se jueguen. No existe corrida automática ni backfill retroactivo de partidos ya jugados. El script run_agent_debate.py es idempotente: acumula nuevos resultados sin pisar debates previos a menos que se use --force.
RN-DEB-03 · ALTA
max_tokens=4500 para consenso con deepseek-reasoner
El deepseek-reasoner cuenta los thinking tokens contra el límite de max_tokens. Con 4 predicciones (3 individuales + 1 consenso) y razonamiento extendido, se requieren mínimo 4500 tokens. Si se agota antes de terminar, el response viene vacío con HTTP 200 — el script debe detectar esto y retry o reportar.
RN-DEB-04 · MEDIA
Agent attribution obligatoria en output
Cada predicción en el JSON debe incluir un campo agent (Group Analyst, Tactical Scout, Sentiment Reader, Consensus) para permitir evaluación granular de precisión por tipo de razonador. Sin atribución, no se puede medir cuál agente es más predictivo.
FinOps & Costos
RN-FIN-01 · CRÍTICA
Fallback graceful al Ensemble cuando el presupuesto se agota
Si el daily_limit, monthly_limit o per_run_limit definidos en budget.yaml se alcanzan, el sistema DEBE continuar operando con el Ensemble determinístico. Nunca bloquear la predicción completa — solo bloquear el enriquecimiento LLM. El usuario recibe predicción, no error.
RN-FIN-02 · ALTA
Logging obligatorio de cada llamada LLM
Cada llamada a cualquier LLM (DeepSeek, Claude) debe generar una entrada en logs/llm_costs.jsonl con: timestamp, agente, modelo, tokens_input, tokens_output, costo estimado, partido. Sin esta información no es posible auditar el gasto real vs presupuestado.
RN-FIN-03 · ALTA
Narrator AI: una llamada LLM por partido por día, máximo
El script precompute_narrations.py genera narraciones una vez al día por partido × dialecto. Nunca re-generar si la clave ya existe en narrations.json del día actual (a menos que sea forzado explícitamente). El LLM en /api/narrator es únicamente fallback para claves faltantes.
RN-FIN-04 · MEDIA
Narrador solo en español bogotano
Todas las narraciones (grupos, knockout, final) se generan exclusivamente en español bogotano. No hay selector de dialecto ni múltiples variantes regionales. Esto simplifica el costo y el mantenimiento: $0.016/partido independientemente de la fase.
Integridad de datos
RN-DAT-01 · CRÍTICA
Todos los JSONs públicos deben ser UTF-8 válido
Cualquier JSON bajo frontend/public/data/ debe escribirse con encoding UTF-8 explícito usando Python (Path.write_text(encoding='utf-8')). Nunca usar PowerShell Get-Content | Set-Content para reescribir estos archivos — produce mojibake (MÃ©xico, arrancÃ³) que rompe el frontend. Test en test_integrity.py.
RN-DAT-02 · CRÍTICA
Fallo silencioso de datos = inaceptable
Todas las aserciones de schema de datos son explícitas y levantan excepciones descriptivas. No se permite que el pipeline continúe con datos malformados o schemas incorrectos. Un DataFrame con NaNs en FEATURE_COLS debe generar error con nombre de columna, no producir predicciones silenciosamente incorrectas.
RN-DAT-03 · ALTA
Modelos y features son regenerables, nunca la fuente de verdad
Los archivos en models/ y data/processed/ están en .gitignore y son completamente regenerables con run_pipeline.py. La fuente de verdad son los scripts Python y los CSV raw. Nunca commitear modelos entrenados — pueden quedar desincronizados con el código.
RN-DAT-04 · ALTA
Secrets nunca en código — solo en variables de entorno
FOOTBALL_DATA_TOKEN, DEEPSEEK_API_KEY, ANTHROPIC_API_KEY y DASHSCOPE_API_KEY deben vivir exclusivamente en .env (local) o en las variables de entorno de Vercel (producción). Los archivos .env están en .gitignore. Ningún test debe requerir keys reales — usar mocks.
Frontend & UX
RN-FE-01 · ALTA
El Ensemble siempre es visible — los deltas son etiquetados
El frontend debe mostrar siempre el prior del Ensemble como la predicción base. Si hay enriquecimiento de agentes, el delta debe estar labeled explícitamente como "enriquecimiento opcional". El usuario nunca debe ver solo la probabilidad final sin contexto de cómo se calculó.
RN-FE-02 · ALTA
Chat IA: rate limit 20 req/hora/IP sin Redis
El API Route de chat implementa rate limiting de 20 requests/hora por IP usando module-level Map con TTL. No se requiere Redis. El límite debe rechazar requests con HTTP 429 y mensaje claro, nunca silenciosamente ignorarlos. Los temas fuera de scope del torneo se rechazan antes de consumir tokens.
RN-FE-03 · MEDIA
Monte Carlo corre 100% en el cliente — nunca en servidor
Las 5,000 simulaciones del torneo se ejecutan en TypeScript en el navegador del usuario. No existe endpoint de servidor para Monte Carlo. Los datos base (probabilidades pre-calculadas por par de equipos) se sirven desde JSON estático. Esto garantiza escalabilidad sin costo de cómputo adicional.
RN-FE-04 · MEDIA
Horarios siempre en zona Colombia/Bogotá (UTC-5)
Todos los horarios mostrados en el dashboard (partidos del día, próximos partidos, ciclo de actualización) deben presentarse en hora de Colombia (UTC-5), independientemente de la zona horaria del usuario. El contexto principal es el espectador colombiano.
RN-FE-05 · MEDIA
Narrador AI: cero llamadas LLM en tiempo real por usuario
El endpoint /api/narrator sirve exclusivamente desde narrations.json pre-computado. Solo se hace una llamada LLM (fallback) si la clave específica (match_id + dialecto) no existe en el archivo. El hit rate objetivo es ≥95%. Un miss rate >5% indica que el ciclo diario no se ejecutó correctamente.
Operaciones & Ciclo diario
RN-OPS-01 · ALTA
Protocolo MD2: doble corrida obligatoria
En Matchday 2, el ciclo completo debe ejecutarse dos veces: una vez por la mañana (antes del primer partido) y una vez por la tarde (después de los primeros 2 resultados). Sin la segunda corrida, los partidos nocturnos se predicen sin contexto de presión real de clasificación generada por los partidos de tarde.
RN-OPS-02 · ALTA
MD3: partidos simultáneos requieren ELO cutoff independiente
En Matchday 3, todos los partidos de un grupo arrancan a la misma hora. El sistema no debe usar el resultado de Partido A para calcular el ELO del Partido B del mismo grupo — ambos tienen el mismo ELO base. Los standings de MD3 son escenarios probabilísticos, no secuenciales.
RN-OPS-03 · MEDIA
El ciclo diario siempre termina con deploy
Ningún paso del ciclo (live_update, predict_live, precompute_narrations) es opcional en producción durante el torneo. El paso 4 (npx vercel --prod) es mandatorio — ejecutar los primeros 3 pasos sin deploy deja el frontend desactualizado y el usuario ve predicciones del día anterior.
RN-OPS-04 · MEDIA
141 tests deben pasar antes de cualquier deploy a producción
Nunca hacer deploy si algún test del suite falla. La suite cubre pipeline, features, modelo, agentes, cost guard, integridad, simulador y agent debate. Tests que fallan indican regresiones potenciales que pueden romper predicciones en vivo. Ejecutar con pytest -v antes de cada deploy manual.

Setup local: paso a paso
1
Clonar el repositorio
El dataset histórico se descarga automáticamente de Kaggle vía kagglehub — los CSV no están versionados en el repo para mantenerlo liviano.
git clone https://github.com/mcoy2025ia/swarm-dataops cd mundial-predictor-2026
2
Crear entorno Python 3.11 e instalar dependencias
Python 3.11 requerido. El entorno virtual evita conflictos con otras instalaciones del sistema.
python -m venv .venv .venv\Scripts\activate # Windows PowerShell # source .venv/bin/activate # Linux / macOS pip install -r requirements.txt
3
Configurar variables de entorno
Crear dos archivos: .env en la raíz (para scripts Python) y frontend/.env.local (para Next.js). Ambos están en .gitignore — nunca commitear.
# .env (raíz — para scripts Python) DEEPSEEK_API_KEY=sk-... ANTHROPIC_API_KEY=sk-ant-... # frontend/.env.local (para Next.js API Routes) FOOTBALL_DATA_TOKEN=... DEEPSEEK_API_KEY=sk-... ANTHROPIC_API_KEY=sk-ant-... DASHSCOPE_API_KEY=... # opcional, para RAG
4
Ejecutar el pipeline completo desde cero
Descarga datos de Kaggle, calcula ELO sobre 49,765 partidos, entrena el Ensemble con split temporal, exporta los JSONs para el frontend. Total: ~90 segundos.
python scripts/run_pipeline.py # raw → ELO → features → modelos → métricas python scripts/export_frontend_data.py # genera JSONs en frontend/public/data/
5
Verificar que los 122+ tests pasan
La suite completa debe pasar antes de continuar. Tests fallidos indican regresiones en el pipeline, features, modelo o contratos de datos.
pytest # todos los tests pytest tests/test_model.py -v # solo modelo pytest tests/test_agents.py tests/test_cost_guard.py -v # agentes + budget pytest tests/test_integrity.py -v # contratos de datos + encoding
6
Instalar dependencias del frontend y levantar en local
Next.js 15 con hot reload. El dashboard completo estará disponible en http://localhost:3000 con todos los módulos funcionales usando los JSONs pre-generados.
cd frontend npm install npm run dev # http://localhost:3000 # Alternativa: Streamlit para inspeccionar el modelo directamente streamlit run src/app.py # http://localhost:8501
7
Deploy a Vercel (producción)
Configurar las env vars en el dashboard de Vercel antes del primer deploy. El CLI toma las variables de Vercel automáticamente en deploys sucesivos. El primer deploy configura el proyecto.
cd frontend npx vercel # primer deploy: configura proyecto + dominio npx vercel --prod # deploys sucesivos a producción
8
Ciclo diario durante el torneo
Ejecutar cada mañana antes de los partidos. En MD2, correr de nuevo por la tarde después de los primeros 2 resultados. Ver pestaña MLOps para el protocolo completo de MD2 y MD3.
python scripts/live_update.py # fetch → ELO → retrain → export python scripts/predict_live.py --export # predicciones + agentes multi-especialista python scripts/precompute_narrations.py # narraciones del día (DeepSeek) cd frontend && npx vercel --prod # deploy
